## Quick Start

```bash
# Development
cd server && npm install && npm run dev   # API on :3000
cd client && npm install && npm run dev   # UI  on :5173 (proxies /api + /thumbnails to :3000)

# Production — Node serves the API and the prebuilt SPA on the same port
docker compose up -d --build              # http://localhost:3000
```

## Key Architecture Notes

- **Single SQLite DB** — `better-sqlite3` with WAL mode, `synchronous = NORMAL`, and a 256 MB page cache + 1 GB mmap window sized for 8 TB libraries. All writes are synchronous on the server thread; no connection pooling needed.
- **Incremental scanning** — The scanner stores each chapter's file mtime in `chapters.file_mtime`. On re-scan, chapters whose mtime is unchanged and already have indexed pages are skipped entirely, making repeated scans of large libraries fast.
- **Parallel scanning** — `scanLibrary` walks manga directories with `MANGA_CONCURRENCY = 4` workers via an internal `withLimit` helper. DB correctness is preserved because `better-sqlite3` serialises writes on a single thread; the speedup comes from overlapping disk I/O.
- **CBZ reads** — Scanning uses `yauzl` to list entries from the archive's central directory (parsed directories are kept in a bounded LRU, mtime-validated, no fd retention). Serving extracts each requested entry to `CBZ_CACHE_DIR/<chapterId>_<mtimeFloor>/<entry>` on first hit and then serves it as a plain file with `res.sendFile`; the extract cache defaults to 20 GB and is **user-configurable at runtime** from Settings → Database. Eviction is **LRU per-chapter** — when an extraction pushes the total over the cap, the cache wipes the least-recently-used chapter directories until the total drops below cap. In-flight extractions are never wiped from underneath themselves. An optional daily/weekly auto-clear schedule fires a total wipe at a user-chosen time (see [scanner.md § CBZ Serve Cache](./scanner.md#cbz-serve-cache)). Streaming serves were swapped for extract-to-disk after concurrent streams were observed cross-contaminating between archives. `pages.path` stores the cache filename for CBZ chapters and the absolute filesystem path for folder chapters; the serving route branches on `chapters.type`.
- **First-page-fast CBZ open** (opt-in per device via Settings → Reading Settings → Advanced) — large CBZ chapters now open after only the first few pages have been extracted, with the remainder extracted in the background by a Phase 2 worker (bounded concurrency, priority-hint queue for scrubber jumps). Phase 1 dim-probes every page via a 256 KB header sniff so Double Page (Manga) layout is correct on first paint; Phase 2 re-probes from the extracted file via `sharp.metadata()` and updates any rows whose dims Phase 1 missed. UPSERT-by-`(chapter_id, page_index)` means `pages.id` is stable across re-extractions — art-gallery entries survive cache evictions. Cancellation (`cancelChapter`) is wired into every chapter-deletion path so a manga removed mid-read aborts in-flight Phase 2 work cleanly; an orphan audit at startup and end-of-`scanLibrary` reaps any cache directories whose chapter rows are gone. See [scanner.md § Fast mode](./scanner.md#fast-mode-first-page-fast).
- **Predictive next-chapter pre-extraction** (default on, migrated from `reader_prefetchPages` on first read; independent thereafter) — when the user is within 3 pages of the end of the current chapter, the reader's prefetch hook calls `api.getPagesWithMeta(next.id, { fast: true })` (when fast-open is also on) so the server starts extracting the next chapter ahead of the navigation. Combined with fast chapter open, navigating to the next chapter lands on a near-instant cache hit.
- **Client-side dim safety net** — every `<img onLoad>` in the reader (displayed pages + prefetched `new Image()` instances) reads `naturalWidth`/`naturalHeight` and reports it via `POST /api/pages/dims` if the page row still has null dims. The browser's native decoder catches any image format `sharp` couldn't read at the server. UPDATE filter `AND (width IS NULL OR height IS NULL)` ensures client reports never overwrite server-probed values. Buffered + batched on the client (16 entries / 800 ms idle). Combined with the defensive-solo rendering of unknown-dim pages in `mangaSpreads`, this means wide spreads in Double Page (Manga) layout can never be visually mispaired regardless of which probe path fails.
- **Cached disk-usage columns** — `manga.bytes_on_disk` / `manga.file_count` are populated at scan time (summed from `chapters.bytes_on_disk` / `chapters.file_count`), so `/api/stats` answers with one query instead of walking the library. (`/api/manga/:id/info` is the exception — it recomputes size and chapter count on demand so its modal, which also reports missing-chapter gaps, reflects the disk's current state rather than the last scan rollup.)
- **Indexed search** — `?search=` no longer scans the manga table. Title/author hits resolve through an FTS5 virtual table (`manga_fts`, `unicode61` tokeniser, whole-word match); genre hits resolve through a normalised `manga_genres(manga_id, genre COLLATE NOCASE)` table with a composite PK. Both structures are maintained by triggers against the canonical `manga.genres` JSON blob and `manga.title`/`manga.author`, so no write path needed to change.
- **Per-device AniList login** — Each browser generates a UUID stored in `localStorage` (`momotaro_device_id`) and sends it as `X-Device-ID` on every request. The server stores AniList tokens in `device_anilist_sessions` keyed by device ID. Logging in on one device does not affect any other.
- **Server-wide Doujinshi.info login** — Doujinshi.info credentials (access token + refresh token) are stored in the `settings` table and shared across all devices, unlike AniList which is per-device.
- **File-first** — Manga and chapter records reflect the filesystem. The scanner upserts rows on every scan. Deleting a manga via the UI removes DB rows **and** the files on disk. The scanner also prunes stale records automatically: chapters missing from disk are removed during `scanMangaDirectory`; manga with no remaining chapters (empty folder, all chapters deleted) are removed at the end of the chapter pass; manga whose folder no longer exists are caught by a library-level cleanup pass at the end of each `scanLibrary` run.
- **AniList sync is fire-and-forget** — `syncToAniList()` runs after the HTTP response is sent. It uses the `X-Device-ID` from the original request to look up the token. Errors are logged but never surfaced to the client. HTTP 429 responses are retried automatically (up to 3 times, honouring the `retry-after` header).
- **All reader settings in localStorage** — No server-side user preferences. Settings persist per browser. The library's default sort order (`home_default_sort`) follows the same pattern and is set from Settings → Homepage Settings; valid values are `title`, `updated`, `year`, `rating`.
- **Rating sort** — `GET /api/library?sort=rating` (and the same parameter on `/api/reading-lists/:id/manga`) orders rows by `m.score DESC NULLS LAST, m.title ASC`, so manga with an AniList or MyAnimeList score rank by that score, and any title that hasn't been matched to a third-party source falls to the bottom alphabetically.
- **Docker data persistence** — `docker-compose.yml` bind-mounts `./data` into the container (not a named volume), so all server state — SQLite DB, thumbnails, CBZ extract cache — lives in the user's filesystem and survives every Docker lifecycle operation including `docker compose down -v`. See [overview.md § Data persistence](./overview.md#data-persistence).
- **Portable config export/import** — `GET /api/admin/export-config` dumps the full user-facing state (settings, API keys, libraries, manga metadata linkage, reading lists, progress, art gallery) to a single JSON file keyed by `manga.path` and chapter `folder_name` instead of auto-increment IDs. `POST /api/admin/import-config` restores the payload inside a single SQLite transaction and remaps everything to the new IDs in the target DB. See [api.md § Configuration Backup](./api.md#configuration-backup).
- **Browse By Genre** — `/genres` renders a grid of every distinct genre across visible libraries. Each tile is decorated with a faded thumbnail of the highest-scored manga in that genre; clicking a tile navigates to `/library` with the genre name pre-filled in the search box, which the existing single-term search route already resolves to a genre filter via the normalised `manga_genres` table — no dedicated filter API was added. The shortcut to the page lives in the sidebar (`AppSidebar`) directly below the Reading Lists section. See [api.md § Genres](./api.md#genres) and [frontend.md § Genres](./frontend.md#genres-srcpagesgenresjsx).
- **Home landing page** — `/` renders a stack of horizontal ribbons (Continue Reading, Discover New Series, Art Gallery, Top Manga in *each* of the top 4 favourite genres) from a single `GET /api/home` fetch. Every ribbon is scoped to libraries visible in the All Libraries view — hiding a library via `show_in_all = 0` removes its manga from Home everywhere. The endpoint has a 30-second in-memory cache and is registered under the service-worker's StaleWhileRevalidate rule, so Home hydrates instantly on every visit. Discover re-seeds client-side on a user-chosen cadence (default daily) so the visible slice of the candidate pool rotates over time without server load. The Art Gallery ribbon auto-rotates via a pure CSS keyframe animation that pauses on hover, focus, touch, off-screen, and `prefers-reduced-motion`. See [frontend.md § Home](./frontend.md#home-srcpageshomejsx).
- **All Libraries statistics respect `show_in_all`** — `GET /api/stats` without a `library_id` scopes every aggregate (`total_manga`, `total_chapters`, `total_pages`, `total_size_bytes`, `total_genres`, `top_genres`, `favorite_genres`, `estimated_read_time_minutes`, `top_manga`) to libraries with `show_in_all = 1` (plus orphan manga with `library_id IS NULL`), matching `/api/home`, `/api/genres`, and `/api/gallery`. Passing `?library_id=N` still returns that single library's totals regardless of its `show_in_all` flag, so the Settings → Statistics dropdown can still inspect a hidden library on demand. Toggling `show_in_all` via `PATCH /api/libraries/:id` invalidates the `/api/stats` and `/api/home` caches so the change is visible on the next fetch instead of after the TTL. See [api.md § Statistics](./api.md#statistics).
- **Independent linkage per source** — a manga can be linked to AniList, MyAnimeList, MangaUpdates, and Doujinshi.info simultaneously. Establishing a new linkage never breaks an existing one; only the explicit Break Linkage action (`POST /api/manga/:id/reset-metadata`) clears IDs. Every metadata-apply path uses `COALESCE` on `anilist_id`/`mal_id`/`mangaupdates_id`/`doujinshi_id` so `null` from the fetched record can never overwrite a stored ID. Display priority for **text fields** is **local > anilist > myanimelist > mangaupdates > doujinshi**: dropping a `metadata.json` switches displayed fields without touching linkages, applying MAL on top of an AniList-displayed manga records the MAL link without changing the visible text, and so on. See [api.md § Linkage and display priority](./api.md#linkage-and-display-priority).
- **Break Linkage falls back to the next-priority remaining source** — `POST /api/manga/:id/reset-metadata` with a `source` that matches the displayed `metadata_source` clears that source's ID/cover and then re-applies from the next remaining linkage in priority order (anilist > mal > mangaupdates > doujinshi). The fallback is cache-first: every prior fetch writes its normalized record to `data/metadata-cache/<source>/<id>.json`, so the common case completes synchronously with no upstream ping; a live fetch only runs when the cache is missing for the chosen source. If no remaining linkage produces a usable record, the row stays at `metadata_source = 'none'` with cleared fields. See [api.md § Fallback after Break Linkage](./api.md#fallback-after-break-linkage).
- **End-of-scan metadata-priority enforcement** — every `scanLibrary` run finishes with a pass that re-aligns every manga's `metadata_source` to the highest-priority remaining linkage, using the same cache-first apply path as the Break Linkage fallback. Drift between linkage IDs and the displayed source (e.g. from older code paths or partial breaks) gets healed automatically without user input. The pass runs *before* cover priority reinforcement so the cover resolver picks the active cover that matches the freshly-chosen display source. See [scanner.md § End-of-scan metadata priority enforcement](./scanner.md#end-of-scan-metadata-priority-enforcement).
- **Per-source metadata cache** — every successful by-ID fetch from AniList, MyAnimeList, MangaUpdates, **and Doujinshi.info** writes its normalized record to `data/metadata-cache/<source>/<id>.json` via [server/src/metadata/cache.js](../server/src/metadata/cache.js). This cache is the source of truth for Bulk Export Metadata (which never re-pings any upstream) and the Break Linkage / scanner-end fallback paths. Doujinshi IDs are URL slugs (TEXT) rather than integers; the cache module sanitizes the filename to keep it inside the source directory regardless of slug shape.
- **Active cover priority is independent of text-field priority** — every fetched cover is saved into its own per-source slot (`anilist_cover`, `mal_cover`, `mangaupdates_cover`, `doujinshi_cover`) and the active `<mangaId>.webp` is chosen by **anilist > mal > mangaupdates > doujinshi > original scan**. Local JSON sidecars don't enter the cover priority — they only swap text fields. A manual user pick (Thumbnail Picker → set-thumbnail) sets `manga.cover_user_set = 1`, which is sticky against subsequent metadata fetches; only the explicit **Reset Thumbnails** op (and the same op that runs at the end of every library scan) clears the flag and re-aligns the cover to the priority order. Reset never pings any upstream — it only re-uses cover files already on disk. See [scanner.md § Cover priority](./scanner.md#cover-priority).
- **Bulk Export reads from the JSON cache** — `POST /api/libraries/:id/export-metadata` (and the per-manga variant) writes `metadata.json` from the highest-priority **previously-fetched** record found in `data/metadata-cache/<source>/<id>.json` (AniList → MAL → MangaUpdates → DB fallback). Export never re-pings any upstream — the data must already have been pulled. The DB row is untouched, so a local-displayed manga keeps showing local fields in the app — only the on-disk sidecar is replaced.
- **Graceful shutdown** — The server handles `SIGTERM` and `SIGINT` by closing the HTTP server cleanly before exiting, with a 10-second force-exit fallback.
- **React 17+ event delegation** — When writing pointer/gesture handlers in React, `e.currentTarget` is the document root, not the DOM element. Always use `useRef` to get the real DOM node for `setPointerCapture` and `getBoundingClientRect`.
